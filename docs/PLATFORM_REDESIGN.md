# Platform Redesign

## 产品定位

这个项目不是“卖图片的 NFT 网站”，而是：

**数字资源所有权证书平台**

资源类型可以包括：

- 小说
- 图片
- 音频
- 视频
- 压缩包
- 其他创作文件

核心规则是：

- 创作者上传资源
- 系统为资源发行一张唯一证书
- 谁持有证书，谁就被视为资源当前拥有者
- 证书交易后，资源所有权也跟着转移

## 当前技术形态

仓库已经收敛成两层：

- 链上层：`Hardhat + Solidity`
- 应用层：`Vite + TypeScript + viem`

这样做的原因很简单：

- 合约逻辑需要标准、稳定、容易部署的工程化工具
- 前端页面少、状态浅，不需要 React 体系带来的额外复杂度

## 链上层设计

核心合约是 `ContentNFTMarketplace`，负责：

- `mint`
- `listForSale`
- `cancelListing`
- `buy`
- `withdraw`
- `withdrawPlatformFees`
- `ownerOf`
- `tokenURI`

链上记录：

- 证书归属
- metadata URI
- 加密内容 URI
- 预览 URI
- 内容摘要 `contentHash`
- 创作者地址
- 版税比例
- 挂牌价格

这里采用的是**非托管挂牌**：

- 挂牌时证书仍然留在卖家钱包里
- 市场只记录价格和卖家
- 购买时合约基于授权完成转移

这样更符合“谁持有证书，谁就是当前拥有者”的产品语义。

## IPFS 与访问控制

如果资源明文上传到 IPFS：

- 任何拿到 CID 的人都可以下载
- NFT 只能证明“谁拥有证书”
- 不能天然保证“只有证书拥有者才能查看内容”

所以现在的方案是：

1. 用户本地先加密资源
2. 加密文件上传到 IPFS
3. metadata 保存 `encryptedContentURI`
4. 当前持证人拿到链下传递的访问密钥包
5. 前端在本地完成解密

这意味着：

- `所有权证明` 在链上
- `解密权限` 仍然是链下分发

## 当前 MVP 范围

现在的最小正式版已经覆盖：

- 创作者上传资源
- 浏览器本地加密
- IPFS 上传 metadata 与加密文件
- 链上发行证书
- 链上挂牌和购买
- 当前持证人在 `My Library` 里下载并本地解密

## 当前边界

这个版本能防住的是：

- 同一资源在平台内重复发行
- 伪造链上归属
- 用本地假商品列表冒充市场

这个版本还不能从物理上防住的是：

- 曾经拿到过明文文件的人私下留存副本
- 解密密钥的完全自动化交接

如果要继续往前做，下一阶段最值得补的是：

1. 链下密钥分发服务
2. 买卖完成后的新密钥交接机制
3. 举报与侵权治理流程
